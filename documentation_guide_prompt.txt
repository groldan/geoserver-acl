# Comprehensive Documentation Guide Prompt

This prompt guides you through creating high-quality technical documentation for a software project using MkDocs, Material theme, and Structurizr diagrams, following best practices established in the geOrchestra Gateway project.

## Phase 1: Understanding the Project

First, analyze the project thoroughly to understand its architecture, components, and purpose:

1. Examine the project structure, focusing on:
   - Core modules and their relationships
   - Main functionalities and features
   - Technology stack and dependencies
   - API interfaces and integration points

2. Review existing documentation:
   - READMEs and existing docs
   - Code comments, especially in key interfaces
   - Issues or discussions related to documentation
   - Architecture diagrams or design notes

3. Identify target audiences:
   - End users
   - System administrators
   - Developers
   - Project contributors

2. Create script for diagram generation (structurizr-generate-diagrams.sh):
```bash
#\!/bin/bash

# Check if structurizr-cli is available
if \! command -v structurizr-cli &> /dev/null; then
    echo "structurizr-cli is required but not installed. Please install it first."
    echo "See: https://github.com/structurizr/cli"
    exit 1
fi

# Base directory
BASE_DIR="$( cd "$( dirname "${BASH_SOURCE[0]}" )" && pwd )"
WORKSPACE_FILE="$BASE_DIR/workspace.dsl"
DYNAMIC_VIEWS_FILE="$BASE_DIR/dynamic-views.dsl"
OUTPUT_DIR="$BASE_DIR/../assets/images/structurizr"

# Create output directory if it doesn't exist
mkdir -p "$OUTPUT_DIR"

# Generate SVG diagrams from workspace.dsl
echo "Generating diagrams from workspace.dsl..."
structurizr-cli export -workspace "$WORKSPACE_FILE" -format plantuml -output "$OUTPUT_DIR"

# Generate SVG diagrams from dynamic-views.dsl if it exists
if [ -f "$DYNAMIC_VIEWS_FILE" ]; then
    echo "Generating diagrams from dynamic-views.dsl..."
    structurizr-cli export -workspace "$DYNAMIC_VIEWS_FILE" -format plantuml -output "$OUTPUT_DIR"
fi

# Convert PlantUML files to SVG
echo "Converting PlantUML to SVG..."
"$BASE_DIR/plantuml-generate-svg.sh"

echo "Diagram generation complete\!"
```

3. Create script for PlantUML conversion (plantuml-generate-svg.sh):
```bash
#\!/bin/bash

# Check if plantuml is available
if \! command -v plantuml &> /dev/null; then
    echo "plantuml is required but not installed. Please install it first."
    echo "See: https://plantuml.com/starting"
    exit 1
fi

# Base directory
BASE_DIR="$( cd "$( dirname "${BASH_SOURCE[0]}" )" && pwd )"
OUTPUT_DIR="$BASE_DIR/../assets/images/structurizr"

# Find all .puml files and convert to SVG
echo "Converting PlantUML files to SVG..."
for puml_file in "$OUTPUT_DIR"/*.puml; do
    if [ -f "$puml_file" ]; then
        echo "Converting $puml_file..."
        plantuml -tsvg "$puml_file"
    fi
done

echo "PlantUML to SVG conversion complete\!"
```

4. Add GitHub workflow for automatic build (.github/workflows/docs.yml):
```yaml
name: Build and Deploy Documentation

on:
  push:
    branches:
      - main
    paths:
      - 'docs/**'
      - 'mkdocs.yml'
      - '.github/workflows/docs.yml'

concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    
    steps:
      - name: Checkout code
        uses: actions/checkout@v3

      - name: Set up Python
        uses: actions/setup-python@v4
        with:
          python-version: '3.10'
          
      - name: Install dependencies
        run: |
          python -m pip install --upgrade pip
          pip install mkdocs mkdocs-material

      - name: Install Structurizr CLI
        run: |
          mkdir -p $HOME/bin
          wget https://github.com/structurizr/cli/releases/download/v1.35.0/structurizr-cli-1.35.0.zip
          unzip structurizr-cli-1.35.0.zip -d $HOME/bin
          echo "$HOME/bin" >> $GITHUB_PATH

      - name: Install PlantUML
        run: |
          sudo apt-get update
          sudo apt-get install -y plantuml

      - name: Generate diagrams
        run: |
          cd docs/structurizr
          chmod +x *.sh
          ./generate-diagrams.sh

      - name: Build MkDocs
        run: mkdocs build

      - name: Deploy to GitHub Pages
        uses: JamesIves/github-pages-deploy-action@v4
        with:
          branch: gh-pages
          folder: site
```

## Phase 5: Creating Architectural Diagrams with Structurizr

Use Structurizr DSL to create clear, consistent diagrams:

1. System Context Diagram (workspace.dsl):
```
workspace {
    model {
        user = person "User" "A user of the system"
        system = softwareSystem "Your System" "Description of your system" {
            tags "YourSystem"
        }
        
        externalSystemA = softwareSystem "External System A" "Description of external system A" {
            tags "External"
        }
        
        externalSystemB = softwareSystem "External System B" "Description of external system B" {
            tags "External"
        }
        
        user -> system "Uses"
        system -> externalSystemA "Fetches data from"
        system -> externalSystemB "Sends data to"
    }
    
    views {
        systemContext system "SystemContext" {
            include *
            autoLayout
            title "System Context diagram for Your System"
            description "Shows the system context of Your System"
        }
        
        styles {
            element "Person" {
                shape Person
                background #08427B
                color #ffffff
            }
            element "YourSystem" {
                background #1168BD
                color #ffffff
            }
            element "External" {
                background #999999
                color #ffffff
            }
        }
        
        theme default
    }
}
```

2. Container Diagram (workspace.dsl):
```
workspace {
    model {
        // System context defined above
        
        // Container level
        system {
            webapp = container "Web Application" "The web application" "Spring Boot" {
                tags "WebApp"
            }
            database = container "Database" "Stores user information" "PostgreSQL" {
                tags "Database"
            }
            api = container "API" "Provides functionality via API" "Spring Boot" {
                tags "API"
            }
            
            webapp -> api "Makes API calls to"
            api -> database "Reads from and writes to"
        }
        
        user -> webapp "Uses"
        api -> externalSystemA "Fetches data from"
        api -> externalSystemB "Sends data to"
    }
    
    views {
        // System context view defined above
        
        container system "Containers" {
            include *
            autoLayout
            title "Container diagram for Your System"
            description "Shows the containers that make up Your System"
        }
        
        styles {
            // Styles defined above
            
            element "WebApp" {
                shape WebBrowser
            }
            element "API" {
                shape Component
            }
            element "Database" {
                shape Cylinder
            }
        }
    }
}
```

3. Component Diagram (workspace.dsl):
```
workspace {
    model {
        // System context and container level defined above
        
        // Component level
        api {
            controller = component "Controller" "Handles API requests" "Spring REST Controller"
            service = component "Service" "Implements business logic" "Spring Service"
            repository = component "Repository" "Handles data access" "Spring Data"
            
            controller -> service "Uses"
            service -> repository "Uses"
            repository -> database "Reads from and writes to"
        }
        
        webapp -> controller "Makes API calls to"
    }
    
    views {
        // System context and container views defined above
        
        component api "Components" {
            include *
            autoLayout
            title "Component diagram for the API"
            description "Shows the components within the API"
        }
    }
}
```

4. Dynamic View Diagrams (dynamic-views.dsl):
```
workspace {
    model {
        // Import the model from workspace.dsl
        \!ref user
        \!ref webapp
        \!ref controller
        \!ref service
        \!ref repository
        \!ref database
    }
    
    views {
        dynamic webapp "UserRegistration" {
            title "User Registration Process"
            user -> webapp "1. Submits registration form"
            webapp -> controller "2. Posts registration data"
            controller -> service "3. Processes registration"
            service -> repository "4. Saves user data"
            repository -> database "5. Inserts record"
            database -> repository "6. Confirms insertion"
            repository -> service "7. Returns result"
            service -> controller "8. Returns status"
            controller -> webapp "9. Returns confirmation"
            webapp -> user "10. Shows success message"
            autoLayout
        }
        
        dynamic webapp "UserAuthentication" {
            title "User Authentication Process"
            user -> webapp "1. Submits login credentials"
            webapp -> controller "2. Posts login data"
            controller -> service "3. Requests verification"
            service -> repository "4. Queries user record"
            repository -> database "5. Executes query"
            database -> repository "6. Returns user data"
            repository -> service "7. Returns user object"
            service -> controller "8. Returns authentication result"
            controller -> webapp "9. Returns session token or error"
            webapp -> user "10. Shows dashboard or error"
            autoLayout
        }
    }
}
```

## Phase 6: Content Creation Guidelines

Follow these guidelines to create consistent, high-quality documentation:

1. General Writing Style:
   - Use clear, concise language
   - Write in present tense
   - Use active voice
   - Target the appropriate audience for each section
   - Maintain consistent technical depth (user guide simpler than developer guide)
   - Always add newlines before lists and code blocks

2. Document Structure:
   - Start each document with a clear title and brief introduction
   - Use consistent heading hierarchy (# for title, ## for major sections, etc.)
   - Include table of contents for longer documents
   - Provide navigation links to related documents

3. Code Examples:
   - Use syntax highlighting with appropriate language tags
   - Keep examples short and focused on the specific concept
   - Include comments where necessary
   - Provide context for where the code fits in the larger system

4. Diagrams:
   - Use Structurizr DSL for architectural diagrams
   - Use Mermaid for simpler flowcharts and sequence diagrams
   - Ensure diagrams have clear titles and legends
   - Keep diagrams focused on illustrating specific concepts

5. Including Configuration Examples:
   - Provide realistic, complete configuration examples
   - Explain key configuration parameters
   - Show both minimal and comprehensive configurations
   - Highlight security-sensitive settings

2. Automation:

3. Feedback Mechanisms:
